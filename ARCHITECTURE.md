# PT 예약 관리 SaaS 아키텍처

이 프로젝트는 PT샵, 필라테스, 헬스장, 개인 트레이너가 매일 사용하는 예약/스케줄 SaaS를 목표로 한다.

## 제품 원칙

- 회원 화면은 로그인 없이 링크 기반으로 예약한다.
- 관리자 화면은 운영자가 빠르게 스캔하고 처리할 수 있어야 한다.
- 예약 중복 방지, 수업권 차감, 링크 재발급은 반드시 서버에서 검증한다.
- 화면은 공통 디자인 시스템과 컴포넌트를 기본 경로로 사용한다.
- 새 기능은 화면 코드에 직접 붙이지 않고 도메인, API, UI 계층 중 맞는 위치에 추가한다.

## 현재 구조

```text
.
├── app.js                  # 화면 조립과 이벤트 바인딩
├── server.js               # HTTP 엔트리, 라우팅, 인증 세션
├── styles.css              # 디자인 토큰과 공통 UI 스타일
├── data/
│   └── pt-booking-data.json
└── src/
    ├── client/
    │   ├── api.js          # 브라우저 API 클라이언트
    │   └── ui.js           # 공통 UI 컴포넌트
    ├── server/
    │   └── storage.js      # 저장소 어댑터
    └── shared/
        └── domain.js       # 공통 도메인 규칙과 포맷터
```

## 계층 규칙

### `src/shared/domain.js`

브라우저와 서버가 함께 쓰는 규칙을 둔다.

- 예약 상태 라벨
- 상태 톤
- 날짜/금액 포맷터
- 시간 슬롯 계산
- 수업 단가 계산

### `src/client/api.js`

화면이 서버 URL을 직접 알지 않게 한다.

- `AdminApi`: 로그인, 로그아웃, 관리자 데이터 로드/저장
- `MemberApi`: 회원 링크 조회, 예약 생성, 예약 취소

### `src/client/ui.js`

반복되는 UI 조각을 둔다.

- 배지
- 지표 카드
- 페이지 제목
- 빈 테이블
- 로딩 화면

### `src/server/storage.js`

저장 방식을 감싼다.

현재는 JSON 파일을 사용하지만, 운영 DB로 바꿀 때는 이 파일을 PostgreSQL/Supabase 어댑터로 교체하는 방향을 우선한다.

## 다음 확장 순서

1. 조직/지점 테넌트 모델 추가
2. 관리자 계정 다중화와 권한 분리
3. JSON 파일 저장소를 PostgreSQL/Supabase로 교체
4. 예약 생성 API에 DB 유니크 제약 추가
5. 알림톡/문자 알림 워커 추가
6. 감사 로그와 데이터 백업 추가
7. 결제/정산 모듈 분리

## 운영 전 필수 체크

- `ADMIN_PASSWORD` 환경변수 설정
- HTTPS 배포
- DB 백업 정책
- 개인정보 처리방침
- 회원 예약 링크 재발급 테스트
- 모바일 카카오톡 인앱 브라우저 테스트
